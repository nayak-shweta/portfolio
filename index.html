The most effective way to resolve conflicting priorities is to show people what their choices actually mean, before those choices create problems. Data does this better than any conversation.
Case Study — Resolving Competing Priorities Across 3 Product Owners
The Situation
Three Product Owners, each with equal priority, each wanting their full feature set delivered within the same quarter. Eight features across two delivery streams competing for the same engineering capacity. Every PO believed their work could run in parallel with everything else.
What I Did
Built a two-scenario delivery timeline — not to argue, but to show. Scenario A modelled everything running in parallel. Scenario B modelled a sequential approach: complete the API migration first, then deliver features on the new pipeline only. The visual made the case no meeting ever could — parallel execution would stretch every feature to July. Sequential got more done, faster.
What Changed
All three Product Owners agreed to the proposed plan, as presented, with no pushback. When people can see the logic for themselves, they make the decision rather than feeling it was made for them. The roadmap was adopted and delivery ran on the faster track.

Chain 1 — AI Test Case Generator
30–60% QA effort reduction · Eliminated seniority gap · Featured in leadership reviews
01
The Problem
Copilot was in use but without context — generating generic steps QAs had to fix almost entirely. Experienced QAs produced thorough cases; juniors didn't. Edge cases were missed. And I was spending disproportionate time reviewing and sending test cases back for rework.
What I Built
A Test Case Generator Agent with one rule — ADO Acceptance Criteria is the only source of truth. Context-aware generation, enforced structure every time, edge cases and negatives built in, preview before commit, ADO-ready output. No reformatting. No copy-paste.
What Changed
QA effort per story dropped 30–60% (40–180 mins saved). Junior QAs now produce cases at the same structural standard as seniors. BA review loop almost entirely eliminated. Time that was going into fixing test cases is now going into actually ensuring quality. Which was the point all along.

Chain 2 — Functional Knowledge Hub Agent
My mini-clone · Team functional independence · 24/7 availability
02
The Problem
I was the team's single point of knowledge — I knew how the system behaved, where the document lived, which ticket had the answer. A question that took someone 45 minutes took me 3. A team dependent on one person for functional knowledge is fragile.
What I Built
A Functional Knowledge Hub Agent — a mini version of me, available 24/7. It actively searches ADO and SharePoint, surfaces relevant features, stories, and design decisions, and synthesises clear answers. Evidence-backed only — never from assumption. Correlates Feature → Story → Test Case → Design Document.
What Changed
The team gained functional independence. Developers understand intent before building. QAs test with confidence. New members onboard faster. Mid-sprint blockers caused by functional uncertainty reduced significantly. And I got time back for work that needs human judgment.
Chain 3 — User Story Generator Agent
50% BA effort saved · Thinking partner · Consistent AC coverage every sprint
03
The Problem
Writing good stories at scale, sprint after sprint, is harder than it sounds. Fatigue sets in — a story that should have five detailed AC gets three. Edge cases get folded into vague catch-alls. And simple technical questions meant waiting for a lead.
What I Built
A User Story Agent that interviews me — asking the questions I might skip. Pulls from ADO feature intent and SharePoint design context. GitHub integration for instant technical validation. AC completeness enforcement — flags thin criteria, missing edge cases, ambiguous scope.
What Changed
Stories are consistently thorough — not just when I'm fresh, but also on heavy sprint days. The agent catches what I might miss. That is not a weakness to hide — that is self-awareness applied productively. 50% of my effort per story goes back into thinking rather than formatting.

Feature Specification — Duplicate Order Form Generation Fix
Sanitised from a real delivery — root cause identified via API analysis, fix scope defined with clear in/out of scope and GIVEN/WHEN/THEN acceptance criteria
Feature ScopingTechnical BARoot Cause Analysis
Business Value & Root Cause
As a subscription platform administrator, I want the order form to regenerate only when cart contents actually change. Root cause (identified via API analysis): The Checkout Validation Service compared contract term values between cart and offer. Modified offer cart line items carry status "Existing" but the Front End sends contract term as a formatted string. The service found a mismatch and issued a re-trigger flag on every download — even with zero cart changes.
✓ In Scope
• Skip contract term check when offer type is Modified
• Apply fix to Automation Pipeline v2 only
• Validate re-trigger flag not generated for modified offers
• Confirm subsequent downloads fetch existing form only
✗ Out of Scope
• Representative-generated offers
• Automation Pipeline v1 offers
• Contract term formatting on Front End
• Other validation flags in the service
Acceptance Criteria
GIVEN Modified Digital offer via v2
WHEN Order Form downloaded
THEN generates once only; subsequent downloads retrieve existing form

GIVEN Modified offer, cart exists
WHEN contract term check runs
THEN re-trigger flag NOT generated; all other flags unaffected

[Service-Retention] — Subscription API: Filter by Delivery Format
Business Value: As a customer, I want the subscription API to return only SaaS-format products, so I see only relevant products without non-applicable formats.

Background: The API currently returns all product types regardless of format. Only SaaS-format products should appear in the subscription interface.

Acceptance Criteria:
GIVEN SaaS products exist → WHEN API called → THEN only SaaS returned (current + future-dated)
GIVEN non-SaaS products exist → WHEN API called → THEN excluded from response

Considerations: Backward compatibility · Format field null handling · Performance impact
✓ Background · Multiple ACs · Edge cases · Considerations · ADO-ready

How I Think — Five Principles
AI should reduce cognitive load, not add process overhead.
Every chain I built started with: what is the most mentally draining part of this task? If automation just moves effort around rather than removing it, it has not solved anything.
Good delivery systems reduce dependency on individuals.
What happens when I'm not here? The answer should be: the same thing as when I am.
Requirements quality determines downstream delivery quality.
My standard: can a developer build this without asking me a single question?
Automate the repetitive decisions, not human judgment.
Mechanical work shouldn't sit on anyone's to-do list. Human judgment is for what remains.
The gap between intent and execution is where projects fail. That is where I work.
Not just documenting what each side says. Translating, challenging, and making sure both sides actually understand each other.
A role where product thinking, AI skills, and delivery experience all matter. Somewhere that values people who build things that make teams better, not just manage them.
